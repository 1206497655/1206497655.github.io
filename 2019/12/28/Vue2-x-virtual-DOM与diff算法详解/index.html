Vue virtual DOM与diff算法详解

前言

虽然网上类似这样的解析vue diff算法的文章到处都是，但是有句话说得好，纸上得来终觉浅，绝知此事要躬行。下面结合源码看看，Vue是怎么通过virtual DOM与diff算法实现高效更新DOM的。篇幅可能较大，如果理解有误的请大佬们多多指教。本文基于VUE2.6.11。

vue如何更新DOM

vue通过virtual DOM与diff算法实现dom的最小更新，而不是直接重绘整个dom，这就节省了许多的性能开销。vue首相先把dom节点抽象成一个oldVnode对象，然后在dom相关的数据被修改之后，生成一个新的vnode对象，此时数据还没真实更新的dom上，而是一边比对vnode与oldVnode，一边把变化通过patch打补丁这个方法更新到真实dom上去，从而实现dom的最小更新，而不是直接重绘整个dom。下面将有一个例子（1）。

virtual DOM是什么？

virtual DOM就是把真实DOM中的属性和数据抽取出来，以对象的方式存储起来的一个对象，这个对象就是virtual DOM的实体。Talk is cheap show me the code.

<div id="div">
    <span id="1">1</span>
    <span id="2">2</span>
    <span id="3">3</span>
</div>

例如上面这个div，对应的虚拟dom结构是以下这样的

const vnode = {
    tag: 'div', //
    text: '',
    elm: '#div', // 对应的真实节点
    children: [ // 子节点对应的vnode
        {
            tag: 'span',
            text: '1',
            elm: '#1',
            children: null
        },
        {
            tag: 'span',
            text: '2',
            elm: '#2',
            children: null
        },
        {
            tag: 'span',
            text: '3',
            elm: '#3',
            children: null
        }
    ]
}

当然上面是简化版的，实际上完整的虚拟dom结构如下：

Diff的整体核心逻辑如下：

patch(oldVnode, vnode)

我们直接通过源码来看看patch()这个方法都做了些什么事（只保留核心代码）

function patch(oldVnode, vnode) {
    // 如果vnode是undefined/null, 且oldVnode是defined, 则表示节点下一步将会被销毁, 
    // 直接执行invokeDestroyHook(oldVnode)销毁真实节点,然后直接return
    if (isUndef(vnode)) {
      if (isDef(oldVnode)) invokeDestroyHook(oldVnode)
      return
    }

    if (isUndef(oldVnode)) {
    // 如果vnode对应的oldVnode不存在, 则表示节点下一步将会被添加，
    // 直接执行createElm(vnode, insertedVnodeQueue)
      createElm(vnode, insertedVnodeQueue)
    } else { 
      // 如果oldVnode存在
      const isRealElement = isDef(oldVnode.nodeType)
      // 如果不是真实元素&&是指向同一个元素的vnode
      if (!isRealElement && sameVnode(oldVnode, vnode)) {
        // patch existing root node   给存在的根节点打补丁
        patchVnode(oldVnode, vnode)
      } else {
        // 如果sameVnode不成立，表明真实节点下一步被其它节点替换，下面代码执行替换操作
        const oldElm = oldVnode.elm // 当前oldVnode对应的真实节点
        const parentElm = nodeOps.parentNode(oldElm) // oldElm的父节点

        // create new node by vnode
        createElm(vnode)

        // 递归更新父节点元素
        // update parent placeholder node element, recursively
        if (isDef(vnode.parent)) {
          let ancestor = vnode.parent
          const patchable = isPatchable(vnode)
          // 如果vnode存在父节点，则先清空现有的子节点，再把vnode对应的节点挂载进去
          if (ancestor) {
            cbs.destroy(ancestor)
            cbs.create( ancestor)
          }
        }
      }
    }
    return vnode
  }

patch()首先根据oldVnode与vnode的存在情况来确定下一步如何处理：

  • vnode不存在，oldVnode存在，表示节点将在下一步被删除。
  • vnode存在，oldVnode不存在，表示节点将在下一步被创建。
  • 不存在vnode与oldVnode皆不存在的情况，若是，则不会执行patch()。
  • vnode与oldVnode二者皆存在，但指向的节点不是同一个节点，表示真实节点在下一步将会被其它节点替换。
  • vnode与oldVnode二者皆存在，指向的节点也是同一个节点，则执行 patchVnode()

patchVnode(oldVnode, vnode)

ok，下面继续看看patchVnode都做了些什么（同样只保留核心代码）

  // vnode打补丁
  function patchVnode(oldVnode, vnode) {
    // 如果vnode没有任何修改，直接return
    if (oldVnode === vnode) {
      return
    }

    const elm = vnode.elm = oldVnode.elm
    const data = vnode.data // 将要更新的数据
    const oldCh = oldVnode.children
    const ch = vnode.children
    // 如果有数据要更新，并且vnode允许patch
    if (isDef(data) && isPatchable(vnode)) {
      for (i = 0; i < cbs.update.length; ++i){
        cbs.update[i](oldVnode, vnode)
      // cbs.update 打补丁实现的集合，包含七种实现，从字面意思即可理解
      // updateAttrs
      // updateClass
      // updateDOMListeners
      // updateStyle
      // update
      // updateDirectives
      }
    }
    if (isUndef(vnode.text)) {     // 如果text===null, 表示当前真实节点不是文本节点
      if (isDef(oldCh) && isDef(ch)) { // oV与v都有子节点
        if (oldCh !== ch) updateChildren(elm, oldCh, ch)
      } else if (isDef(ch)) { // 仅v有子节点
        addVnodes(elm, null, ch)
      } else if (isDef(oldCh)) { // 仅oV有子节点
        removeVnodes(oldCh, 0, oldCh.length - 1)
      } else if (isDef(oldVnode.text)) { // 仅oV有text
        nodeOps.setTextContent(elm, '')
      }
    } else if (oldVnode.text !== vnode.text) { // 如果真实节点是文本节点并且文本被更新
      nodeOps.setTextContent(elm, vnode.text) // 直接更新文本
    }
  }

在path()方法中，确定oldVnode与vnode都指向同一个节点后，patchVnode开始对oldVnode与vnode做了以下这几件事：

  • 如果vnode没有任何修改，直接return
  • 如果vnode有数据要更新，并且vnode允许patch，则对真实节点进行更新。
  • 如果真实节点是文本节点且文本得到更新，则直接更新真实节点的文本节点。
  • 如果vnode.text不存在（表明真实节点不是文本节点），则再根据进行情况细分，情况一：如果oldVnode与vnode都有子节点并且不相同，则执行updateChildren；情况二：如果仅vnode有子节点，则直接把子节点挂载至真实节点；情况三：如果仅仅只有oldVnode有子节点，则直接把真实节点的子节点删除；情况四：如果仅vnode有text，则直接把text设置至真实节点。

以上就是patchVnode的内部动作（再回头看看Diff算法流程图），除了updateChildren这一步，其它都比较容易理解，下面继续看看updateChildren做了些什么。

updateChildren

下面通过源码继续对此方法进行解析，同样先贴出源码，此方法逻辑较为复杂难以理解，如果你能理解例子，再回来看看代码就简单多了。

首先先弄明白这个方法的做了什么：

  • 将Vnode的子节点newCh和oldVnode的子节点oldCh提取出来
  • oldCh和newCh各有两个头尾的指针StartIdx和EndIdx，它们的2个指针相互比较，一共有4种比较方式。如果设置了key，就会用key进行比较，在比较的过程中，指针会往中间靠，一旦StartIdx>EndIdx表明oldCh和newCh至少有一个已经遍历完了，就会结束比较。
  function updateChildren(parentElm, oldCh, newCh, insertedVnodeQueue, removeOnly) {
    let oldStartIdx = 0 // oldVnode开始patch的首子节点的指针
    let newStartIdx = 0 // vnode开始diff的首子节点的指针
    let oldEndIdx = oldCh.length - 1 //oldVnode开始patch的末子节点的指针
    let oldStartVnode = oldCh[0] // oldVnode首个子vnode
    let oldEndVnode = oldCh[oldEndIdx] // oldVnode的末尾子vnode
    let newEndIdx = newCh.length - 1 // 执行完一次patch后vnode下一个末尾元素指针
    let newStartVnode = newCh[0] // 执行完一次patch后vnode下一个首元素指针
    let newEndVnode = newCh[newEndIdx] //执行完一次patch后vnode下一个末尾子节点
    let oldKeyToIdx, idxInOld, vnodeToMove, refElm

    // 一直循环直至oldVnode或者vnode的StartIdx>EndIdx
    while (oldStartIdx <= oldEndIdx && newStartIdx <= newEndIdx) {
      if (isUndef(oldStartVnode)) {
        oldStartVnode = oldCh[++oldStartIdx] // Vnode has been moved left
      } else if (isUndef(oldEndVnode)) {
        oldEndVnode = oldCh[--oldEndIdx] // 指针往左移动
      } else if (sameVnode(oldStartVnode, newStartVnode)) { // 如果是指向同一个vnode，则patch
        patchVnode(oldStartVnode, newStartVnode, insertedVnodeQueue, newCh, newStartIdx)
        oldStartVnode = oldCh[++oldStartIdx] // 指针往右移动
        newStartVnode = newCh[++newStartIdx] // 指针往右移动
      } else if (sameVnode(oldEndVnode, newEndVnode)) {
        patchVnode(oldEndVnode, newEndVnode, insertedVnodeQueue, newCh, newEndIdx)
        oldEndVnode = oldCh[--oldEndIdx] // 指针往左移动
        newEndVnode = newCh[--newEndIdx] // 指针往左移动
      } else if (sameVnode(oldStartVnode, newEndVnode)) { // Vnode moved right
        patchVnode(oldStartVnode, newEndVnode, insertedVnodeQueue, newCh, newEndIdx)
        oldStartVnode = oldCh[++oldStartIdx] // 指针往右移动
        newEndVnode = newCh[--newEndIdx] // 指针往左移动
      } else if (sameVnode(oldEndVnode, newStartVnode)) { // Vnode moved left
        patchVnode(oldEndVnode, newStartVnode, insertedVnodeQueue, newCh, newStartIdx)
        oldEndVnode = oldCh[--oldEndIdx] // 指针往左移动
        newStartVnode = newCh[++newStartIdx] // 指针往右移动
      } else {
        // 先根据oldCh生成关于key的hash表
        if (isUndef(oldKeyToIdx)) oldKeyToIdx = createKeyToOldIdx(oldCh, oldStartIdx, oldEndIdx)
          // 判断vnode的newStartVnode有没有key，如果有则到hash表中进行匹配返回idx，如果没有则key的话, 则拿newStartVnode去oldCh里遍历查看有无sameVnode，有的话返回idx，没有则返回null
        idxInOld = isDef(newStartVnode.key)
          ? oldKeyToIdx[newStartVnode.key]
          : findIdxInOld(newStartVnode, oldCh, oldStartIdx, oldEndIdx)
        if (isUndef(idxInOld)) { // 如果idxInOld为null，则表明是新元素，直接创建新元素即可
          createElm(newStartVnode, insertedVnodeQueue, parentElm, oldStartVnode.elm, false, newCh, newStartIdx)
        } else {
            // 如果idxInOld不为null则进行patch
          vnodeToMove = oldCh[idxInOld]
          if (sameVnode(vnodeToMove, newStartVnode)) {
            patchVnode(vnodeToMove, newStartVnode, insertedVnodeQueue, newCh, newStartIdx)
            oldCh[idxInOld] = undefined
          } else {
            // same key but different element. treat as new element
            createElm(newStartVnode, insertedVnodeQueue, parentElm, oldStartVnode.elm, false, newCh, newStartIdx)
          }
        }
          // 指针往右移动
        newStartVnode = newCh[++newStartIdx]
      }
    }
    // 循环比对完成，处理后续工作
    if (oldStartIdx > oldEndIdx) {
      refElm = isUndef(newCh[newEndIdx + 1]) ? null : newCh[newEndIdx + 1].elm
      addVnodes(parentElm, refElm, newCh, newStartIdx, newEndIdx, insertedVnodeQueue)
    } else if (newStartIdx > newEndIdx) {
      removeVnodes(oldCh, oldStartIdx, oldEndIdx)
    }
  }

ps：

（1）oldStartIdx => oS，oldEndIdx => oE，newStartIdx => nS，newEndIdx => nE。

（2）节点皆带有key值。

上面的oldCh[oS]、oldCh[oE]将会与newCh[nS]、newCh[nE]进行交叉比较，也就是前面所说的四种情况：

  • oldCh[oS].key === newCh[nS].key ？
  • oldCh[oS].key === newCh[nE].key ？
  • oldCh[oE].key === newCh[nS].key ？
  • oldCh[oE].key === newCh[nE].key ？

根据以上匹配原则，我们得到oldCh[oS=0].key === newCh[nS=0].key成立，则对应的真实dom 刷牙不需要移动位置，执行patchVnode（没错，递归操作）更新内容，然后oS++、nS++。继续得到如下这样的图示：

继续按照上面四种原则匹配，得到oldCh[oE=2].key === newCh[nS=1].key成立，则需要把对应的真实dom吃早餐移动到第二位（对应newCh中吃早餐排第二的位置），也就是插在刷牙和洗脸中间执行patchVnode，并且oE–、nS++，继续得到下面的图示：

继续执行匹配，得到oldCh[oS=1].key === newCh[nS=2].key与oldCh[oE=1].key === newCh[nS=2].key成立（其实只要oS===nS，oE===nS就不会执行，因为else if只会执行一个），则需要把真实dom洗脸移动到末尾，对应newCh的位置，因为在上一次调整吃早餐dom的时候，间接地把洗脸dom移动到了末尾，所以此次匹配不再需要移动，直接进行patchVnode，并且oS++、nS++（oE不会进行--操作，因为oE===nS没有执行），得到下图：

此时发现oS > oE，即while (oldStartIdx <= oldEndIdx && newStartIdx <= newEndIdx)已经不成立，本次循环结束，开始处理后续工作：

if (oldStartIdx > oldEndIdx) {
    refElm = isUndef(newCh[newEndIdx + 1]) ? null : newCh[newEndIdx + 1].elm
    addVnodes(parentElm, refElm, newCh, newStartIdx, newEndIdx, insertedVnodeQueue)
} else if (newStartIdx > newEndIdx) {
    removeVnodes(oldCh, oldStartIdx, oldEndIdx)
}

简单概括下，循环结束时，如果是因为oS>oE结束的，表明newCh中存在有新加入的vnode，需要执行addVnodes操作（创建对应的真实dom至父节点中），相反，如果是因为nS>nE结束的，表明newCh中删除了某些vnode，则需要执行removeVnodes操作（从父节点中删除vnode对应的节点）。

总结

此次研究收获还是挺大的。不过值得吐槽的是，无论我的节点是动态节点还是静态节点，只要数据一更新，所有的节点都会被递归diff一遍，不得不说也是一个不妙的情况，理应是仅仅对动态节点进行diff的。怪不得vue3.0会对diff算法进行大刀阔斧的改动。
